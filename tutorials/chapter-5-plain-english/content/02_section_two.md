# 2. Your Search Engine Already Knows More Than You Think

Here's the surprise this section is built around: the search engine you may already have is, in a meaningful sense, a knowledge graph database. You don't need to spin up Neo4j. You don't need to pay for a separate system. The graph is already there, hiding in plain sight inside the engine.

To see why, we need to look at what's actually inside a search engine. Most explanations skip this part. We won't.

## What's inside a search engine

Imagine you have a library of ten thousand documents. Forum posts, product descriptions, support tickets — pick anything. To search them quickly, the engine builds two reference structures while it ingests the documents. Just two.

### The inverted index

The first one is called the *inverted index*. It answers the question: *for each unique term that appears anywhere in my documents, which documents contain it?*

If your collection were a paper library, this is the index in the back of every book — but applied across the whole library. Look up *advil*, and you get a list of every document that mentions advil. Look up *headache*, and you get a list of every document that mentions headache.

[FIGURE: fig03_inverted_index]

The inverted index is what makes search fast. When you type *advil* in the search box, the engine doesn't read through ten thousand documents looking for the word. It looks the word up in the inverted index, gets back a list of document IDs, and returns those. Same idea as the index in the back of a book — flip to the page, don't read the whole book.

### The forward index

The second structure goes the other direction. It's called the *forward index* — sometimes called the *uninverted index* in older texts, because it's the reverse of the inverted one. It answers: *for each document, which terms appear in it?*

[FIGURE: fig04_forward_index]

If the inverted index is the index in the back of a book, the forward index is the table of contents — for each chapter, here are the words. Same information, organized the other way around.

Most search engineers know about the inverted index. The forward index gets less attention because most queries don't need it. But it turns out to be the secret to turning a search engine into a knowledge graph.

> **Technical Stuff.** In Apache Solr and similar engines, the forward index is sometimes called "doc values" — values stored per document that you can quickly look up by document ID. The same information was historically used for sorting and faceting, where you need to know what values appear in matching documents. You can skip the terminology, but if you ever see "doc values" in a search engine's documentation, this is what they're talking about.

## Two indexes, one graph

Here's the move that turns this into something graph-like.

If you have the inverted index, you can go *from a term to a set of documents*. You start with *advil*, and you get back doc 12, doc 47, doc 92.

If you have the forward index, you can go *from a document to a set of terms*. You start with doc 12, and you get back the terms it contains — *advil*, *headache*, *dose*.

Put those together and you can do a two-step walk. Start with a term. Go to the documents containing it. Then go to all the *other* terms in those documents.

[FIGURE: fig05_two_step]

That's a graph traversal. The starting term is one node. Each document is a node. Each other term is a node. The "appears in" relationship is an edge. You walked from one node, through some intermediate nodes, to a set of destination nodes.

We just walked a graph using nothing but a search engine.

> **Remember.** A graph traversal in this setup is two index lookups in disguise. Inverted index gets you from a term to documents. Forward index gets you from those documents to other terms. Composed, they let you walk through your content as if it were a graph.

This is the whole secret. Once you have both indexes — and any modern search engine does — you have the bones of a knowledge graph for free.

## Why this matters

So what? Why is this worth a section?

Because building and maintaining a separate graph database is expensive. It's a second system to deploy, monitor, back up, secure, scale, and keep in sync with whatever else has your content. Lots of companies that started this journey by buying a dedicated graph database have eventually realized that for many use cases, the search engine they already had was enough.

In particular, *autogenerated* knowledge graphs — the kind we're building in this chapter — work especially well in a search engine. Every time you add a new document, the indexes automatically update. The graph updates with them. No separate ETL job. No nightly batch. No graph rebuild.

> **Tip.** This doesn't mean a dedicated graph database is never the right tool. If your graph is mostly hand-curated, has rich and consistent edge types, and is queried in deeply nested ways (think six-step traversals through typed relationships), a real graph database may pay for itself. But if your graph is implicit in your text and your queries are one or two steps deep — the most common case — the search engine you have is plenty.
